# TransAI v2.6.4

## Исправления сохранения AI-аналитики (DeepSeek)

1. **Структура таблицы `ai_reasoning`** — в `schema_ai_reasoning.sql` уникальный ключ стоял на `project_id` (NOT NULL, без значения по умолчанию), тогда как API сохраняет данные по `rm_id`. В strict-режиме MySQL сохранение падало с ошибкой; в нестрогом — все строки записывались с `project_id=0` и затирали друг друга через `ON DUPLICATE KEY UPDATE`. Теперь: `UNIQUE KEY uq_rm_id (rm_id)`.
2. **`api/migrate_v2.6.4.sql`** — идемпотентная миграция для существующих установок: удаляет дубли по `rm_id`, убирает ошибочный ключ `uk_project`, делает `project_id` nullable, добавляет `uq_rm_id`. Безопасна для повторного запуска.
3. **`api/ai_reasoning.php`** — пустой вердикт больше не отклоняется (fallback «Не определён»).
4. **`api/upload.php`** — повторная загрузка файлов РМ+ДБ без файла ИИ-аналитики больше не затирает сохранённые DeepSeek-вердикты (`CASE WHEN :has_ai = 1 THEN ... ELSE ai_verdict END`).
5. **`src/components/AIAnalysisPanel.tsx`** — ответ сервера при сохранении теперь проверяется; ошибки выводятся пользователю (ранее «Сохранено в БД» показывалось даже при сбое).
6. **`api/ping.php?db=2`** — диагностика структуры таблиц `ai_reasoning`, `ai_analysis`, `projects`.

## Развёртывание

1. Загрузить на хостинг файлы из `TransAI-v2.6.4-hosting.zip` с заменой (включая скрытые `.htaccess`).
2. **Обязательно** выполнить `api/migrate_v2.6.4.sql` в phpMyAdmin — без этого сохранение аналитики не заработает.
3. Проверить: `https://<сайт>/api/ping.php?db=2` — у таблицы `ai_reasoning` должен быть индекс `uq_rm_id` по колонке `rm_id`.
4. Запустить DeepSeek-анализ на любом проекте — должно появиться «Сохранено в БД»; при ошибке отобразится её текст.
